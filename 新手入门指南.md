## 三、项目概述

### 3.1 这个项目是做什么的？（一句话）

一个 **AI 驱动的计算机科学学习平台**：学生可以通过知识图谱可视化探索学科知识、用自然语言向AI助教提问、做练习题并得到归因分析、获得个性化学习计划；教师可以管理班级、查看学情报告、获得智能预警。

### 3.2 阶段性目标

| 阶段 | 版本 | 目标 |
|------|------|------|
| **第一期（当前）** | v0.1.0 → v1.0 | 完成双引擎架构（助学+助教），实现核心功能闭环：用户系统、GraphRAG问答、知识图谱可视化、错题分析、学情报告。聚焦 **408考研 + 数据库原理** 两大场景 |
| **第二期** | v1.1 | 模型微调（LoRA/SFT）、算法刷题模块（在线编码评测）、移动端适配 |
| **第三期** | v2.0 | 多学科扩展（电子、通信等理工科）、社交学习功能 |

### 3.5 核心功能的数据流（数据怎么跑的？）

#### 流1：「快速问答」— **已实现**（SSE 流式输出）

```
用户在 Chat.vue 输入问题
    │
    ▼
useChat.sendMessage()                # composable 收集历史消息
    │
    ▼
sendQuickMessage()                   # api/ai.ts 发起 fetch 请求
    │ POST /api/ai/chat/quick        # 携带 JWT token + {message, history}
    ▼
Backend: ai_gateway.py               # 收到请求
    │ /api/ai/chat/quick 路由
    │ 添加 X-Service-Token 头
    │ httpx.stream → 转发到 AI 服务
    ▼
AI Engine: rag.py                    # 收到请求
    │ /rag/query/stream 路由
    │ verify_service_token() 校验身份
    │ 调用 LLMClient.stream()
    ▼
LLMClient.stream()                   # engines/llm/client.py
    │ 连接 vLLM (本地 Qwen3.5-9B)
    │ 逐 token 返回流式文本
    │ yield "data: {json}\n\n"       # SSE 格式
    ▼
SSE 流 → Backend 透传 → 浏览器
    │
    ▼
useChat 逐 chunk 更新 messages       # 界面实时显示回答
ChatMessage.vue 渲染 Markdown+LaTeX  # 格式化展示
```

**关键中间件链路**：
```
浏览器 ←[SSE]→ Backend(ai_gateway.py) ←[SSE+ServiceToken]→ AI(rag.py) ←[SSE]→ LLM(vLLM)
```

#### 流2：「知识图谱检索增强问答」— **框架已搭，逻辑待实现**

```
用户提问
    │
    ▼
Backend: /api/ai/chat/deep 路由
    │
    ▼
AI Engine: /rag/query 路由
    │
    ├── GraphRetriever        # 从 ChromaDB 检索相关知识点
    │   └── ChromaDB 向量检索  # 找到语义最相近的知识片段
    │
    ├── RAGGenerator          # 拼提示词模板（prompts.py）
    │   └── RAG_SYSTEM_PROMPT # "你是一个CS助教，基于以下知识片段回答..."
    │
    └── LLMClient.chat()      # 调用远程大模型生成答案
        └── 返回: answer + sources（引用来源）
```

#### 流3：「用户登录注册」— **框架已搭，逻辑待实现**

```
用户填写表单 → Login.vue
    │
    ▼
useAuth.login()              # composables/useAuth.ts
    │ POST /api/auth/login  {username, password}
    ▼
Backend: auth.py             # /api/auth/login 路由
    │
    ├── auth_service.py      # 查数据库，验证密码
    ├── security.py          # hash_password() / verify_password()
    └── security.py          # create_access_token() → 返回 JWT
    │
    ▼
前端: auth store             # stores/auth.ts
    │ localStorage.setItem('token', token)  # 保存 token
    │
    ▼
后续请求: request.ts 拦截器  # 自动在请求头注入 Authorization: Bearer <token>
```

#### 流4：「GNN 题目推荐」— **框架已搭，逻辑待实现**

```
学生答题数据
    │
    ▼
Backend: /api/ai/recommend

    │
    ├── GNNInference         # engines/gnn/inference.py
    │   └── KnowledgeGNN     # 加载训练好的异构图模型
    │   └── 输出: 学生薄弱知识点 + 推荐题目列表

APScheduler (main.py lifespan 启动)

    ├── 每天 04:00 → daily_profile     # 更新学生画像

    │
    ▼

    │
    ▼

#### FastAPI — Python 异步 Web 框架

| 维度 | 说明 |
|------|------|
| **它是什么** | 一个**超高性能的 Python Web 框架**，专门用于构建 RESTful API。它最大的特色是：①自动生成交互式 API 文档（Swagger UI），访问 `/docs` 即可在线测试每个接口；②基于 Python 类型注解自动做数据校验；③原生异步支持，性能接近 Node.js。 |

**一个小对比帮理解**：
```
传统后端 (Flask/Django):
   请求 → 同步处理（等待数据库返回期间线程闲置）
   QPS: ~500

FastAPI:
   请求 → 异步处理（等待数据库时去处理别的请求）

| 维度 | 说明 |

#### SQLAlchemy 2.0 + asyncpg — 异步 ORM

| 维度 | 说明 |
|------|------|
| **它是什么** | SQLAlchemy 是 Python 最强大的 **ORM（Object-Relational Mapping，对象关系映射）**。它让你用 Python 类来操作数据库表，不用手写 SQL 语句。`asyncpg` 是 PostgreSQL 的高性能异步驱动，让数据库操作不阻塞主线程。 |
| **应用场景** | 所有需要关系型数据库（PostgreSQL/MySQL）的 Python 后端。 |
| **项目中的位置** | `core/database.py` 创建 async SQLAlchemy 引擎；`models/` 目录下定义 ORM 模型类（一个类 = 一张表）；`services/` 中用 ORM 操作数据库。 |

**对比手写 SQL vs ORM**：
```python
# 手写 SQL（繁琐、易出错、SQL注入风险）
result = await db.execute("SELECT * FROM users WHERE email = $1", [email])

# ORM 写法（类型安全、自动防注入）
result = await db.execute(select(User).where(User.email == email))
```

| 维度 | 说明 |
|------|------|
| **它是什么** | SQLAlchemy 配套的**数据库版本管理工具**。当你的数据表结构需要变更（比如给用户表加一个 `avatar_url` 字段），Alembic 帮你自动生成 SQL 变更脚本并记录版本，可以方便地升级/回滚。 |
| **应用场景** | 所有使用 SQLAlchemy 且需要迭代表结构的项目。 |
| **项目中的位置** | `alembic.ini` 配置文件。 |

#### python-jose — JWT 认证

| 维度 | 说明 |
|------|------|
| **它是什么** | JWT（JSON Web Token）的 **Python 实现库**。JWT 是一种"无状态认证"方案：用户登录成功后，服务端签发一个加密的令牌（token），后续请求只需携带这个 token 就能证明身份，服务端不用维护 session。 |
| **应用场景** | API 认证、单点登录（SSO）、微服务间认证。 |
| **项目中的位置** | `core/security.py` 中的 `create_access_token()` 和 `decode_access_token()`；`core/deps.py` 中的 `get_current_user()` 依赖注入。 |

**JWT 工作流程**：
```
登录 → 后端验证密码 → 签发 JWT(含用户ID+角色+过期时间) → 前端存 localStorage
     → 后续每个请求头带 Authorization: Bearer <JWT>
     → 后端 deps.py 自动解码验证 → 注入当前用户信息
```

| 维度 | 说明 |

| **应用场景** | 所有需要用户密码的系统。**绝对不要明文存密码！** |
| **项目中的位置** | `core/security.py` 中 `hash_password()`（注册时调用）和 `verify_password()`（登录时调用）。 |

#### httpx — 异步 HTTP 客户端

| 维度 | 说明 |

| **应用场景** | 微服务间通信、代理转发、调用第三方 API。 |
| **项目中的位置** | `backend/app/api/ai_gateway.py` 用 httpx 的流式客户端将前端请求**透传**到 AI 引擎；`ai/app/engines/llm/client.py` 用 httpx 连接 LLM API。 |

**传统搜索 vs 语义搜索**：
```
传统搜索: "数据库范式" → 匹配包含"数据库"和"范式"两个词的文档
语义搜索: "数据库范式" → 也匹配到"关系模式的规范化理论"，因为意思相近
```

#### PyTorch + PyTorch Geometric — 深度学习框架

| 维度 | 说明 |
|------|------|
| **它是什么** | PyTorch 是 Meta（Facebook）开源的**深度学习框架**，用来构建和训练神经网络。PyTorch Geometric（PyG）是其扩展，专门处理**图结构数据**（节点+边），可以实现图神经网络（GNN）。 |
| **应用场景** | 图像识别、自然语言处理、推荐系统、知识图谱推理、分子结构预测等。 |
| **项目中的位置** | `ai/app/engines/gnn/` 下三个文件——`model.py`（定义异构图神经网络模型）、`trainer.py`（训练模型）、`inference.py`（用训练好的模型做推理推荐）。GNN 用于：基于学生-知识点-题目的异构图关系，推荐最适合学生当前水平的练习题。 |

**为什么 GNN 适合这个场景？**
```
学生 ──[做了]──→ 题目 ──[属于]──→ 知识点
学生 ──[不会]──→ 知识点（薄弱点）
        ↓
GNN 可以通过图结构传播信息：
"学生A和B都不会操作系统，但A会数据结构 → 推荐A先复习数据结构 →
再学操作系统，因为两者有关联"
```

## 八、代码架构设计模式（学到什么）

本项目使用了几种重要的设计模式，理解它们有助于快速上手：

### 8.1 分层架构（Backend）

```
api/ (路由层) → services/ (业务逻辑层) → models/ (数据层)
         ↕
    schemas/ (数据校验层)
         ↕
    core/ (基础设施层——配置/数据库/安全)
```

**好处**：每层职责清晰，修改数据库不影响业务逻辑，修改业务逻辑不影响 API 接口。

### 8.2 装饰器自注册模式（AI 任务系统）

```python
# 新增一个定时任务只需：
@scheduled_task("my_task", trigger="cron", hour=8, minute=0)
async def my_daily_task():
    ...

# 无需修改 scheduler.py 或 worker.py —— 自动发现并注册
```

**好处**：新增任务时不用修改核心代码，符合"开闭原则"（对扩展开放，对修改关闭）。

### 8.3 依赖注入模式（FastAPI Depends）

```python
@router.get("/me")
async def get_me(user = Depends(get_current_user)):
    # get_current_user 自动从请求头解析 JWT、查数据库、返回用户
    # 路由函数只需声明依赖，不需要写鉴权代码
```

**好处**：鉴权逻辑写一次，所有路由复用。测试时可以轻松替换依赖。

### 8.4 Deep Module 模式（LLMClient）

```python
# 调用方只需要：
llm = LLMClient(default_profile=quick_profile())
answer = await llm.chat(messages)

# 内部隐藏了：HTTP连接管理、auth注入、重试、超时、异常处理
```

**好处**：复杂逻辑隐藏在模块内部，对外接口极简。换一个 LLM 服务只需换一个 profile，不改业务代码。

## 十、常见问题

**Q: 为什么有三个服务而不是一个？**
A: 这是"微服务"思想——前端的 UI 逻辑、后端的业务逻辑、AI 的模型推理各自独立，可以分别扩容、分别部署、分别出问题不影响其他部分。

**Q: 为什么用 Vue 而不用 React？**
A: Vue 的中文生态更好，学习曲线更平缓，适合学生团队和国内场景。

**Q: 为什么 Backend 和 AI Engine 不合并？**
A: AI 引擎需要 GPU 资源（PyTorch），是很"重"的服务；Backend 是轻量级的 CRUD。分开部署可以让 Backend 跑在便宜的 CPU 服务器上，AI 引擎跑在 GPU 服务器上。

**Q: 为什么用 SSE 而不是 WebSocket？**
A: 聊天是"一问一答"模式，SSE 足够（单向推送），比 WebSocket 更简单、更省资源。WebSocket 适合需要双向实时通信的场景（如在线协作编辑）。

**Q: 桩代码（stub / `pass`）是什么意思？**
A: 函数或类已经定义了（名字、参数、返回值都已确定），但具体实现还没写，用 `pass` 占位。这是"先搭骨架再填充"的开发方式——架构和接口设计先行，功能逐步实现。
