## 一、先给结论

### 1.1 这个项目现在到底是什么

这是一个面向 **计算机科学学习场景** 的 Web 全栈项目，目标是做成“学生侧助学 + 教师侧助教”的 AI 平台。  
当前代码仓库已经形成了 **前端 + 后端 + AI 引擎** 三层结构，但真正打通并有明确代码实现的，主要是下面三条链路：

1. **AI 助教聊天**
   - 前端支持“快速回答”和“深度思考”两种聊天模式
   - 后端负责转发
   - AI 服务负责流式调用模型并通过 `SSE`（服务端流式推送）返回

2. **OCR 文档解析**
   - 这里的 `OCR` 指文档文字识别/文档解析
   - 前端可以上传 `PDF` / 图片
   - 后端把文件转发给 AI 服务
   - AI 服务内部用 `mineru` 跑异步 `OCR` 任务

3. **知识图谱构建与浏览**
   - 前端可以上传文档发起图谱构建任务
   - 后端维护图谱元数据和任务状态
   - AI 服务执行“`OCR` -> Markdown 切片 -> `LLM` 抽取实体关系 -> NetworkX 合并 -> Apache `AGE` 落库”
   - 前端可以查询图谱列表、浏览图谱、搜索节点、删除图谱

### 1.2 当前项目的真实完成度

按“代码已经实现”而不是“PRD 规划”来判断，当前状态更准确地说是：

| 模块 | 当前状态 | 说明 |
|------|------|------|
| 前端整体框架 | 已搭好 | 路由、布局、状态管理、`API` 分层基本齐全 |
| 聊天功能 | **可用** | 快速回答、深度思考两条 `SSE` 聊天链路都有代码 |
| OCR 解析 | **可用** | 前后端和 AI 任务管理链路都已实现 |
| 知识图谱构建 | **主链路已实现** | 构建任务、状态轮询、`AGE` 写入、图谱可视化查询已具备 |
| 图谱元数据管理 | **可用** | PostgreSQL 中已有 `kg_graphs`（`kg = knowledge graph`，知识图谱）表和对应服务 |
| 登录注册 | 仅接口骨架 | 前端表单页是占位，后端 auth 路由是 `pass` |
| 学习计划/题库/错题本/教师端 | 大多为占位 | 页面、路由、接口名已定义，但业务逻辑未落地 |
| GraphRAG 检索增强问答 | 仅骨架 | `GraphRAG` 所需提示词、类结构有了，检索/生成主逻辑没写完 |
| GNN 推荐 | 仅骨架 | `GNN` 依赖和接口预留了，但模型、训练、推理都没实现 |

### 1.3 不要被旧文档误导的几个关键事实

下面这些点，以当前代码为准：

1. **前端不只是聊天页了**
   - 现在前端已经有：
     - `Chat.vue`
     - `OcrParse.vue`（`OCR` 解析页面）
     - `KgPipeline.vue`（`Kg = Knowledge Graph`，知识图谱构建流水线页面）
     - `KnowledgeExplore.vue`（知识图谱浏览页面）
   - 其中后三者都不是纯占位页。

2. **后端不只是 `auth / knowledge / learning / practice / teacher / ai_gateway` 六个路由**
   - 实际还注册了：
     - `ocr_gateway`
     - `kg_gateway`

3. **AI 服务也不只是聊天**
   - 实际还包含：
     - `ocr/`
     - `kg_pipeline/`（`kg = knowledge graph`，知识图谱流水线模块）
     - `tasks/kg_build.py`（`kg = knowledge graph`，知识图谱构建任务）

4. **当前图谱存储真正用到的是 Apache AGE，不是 ChromaDB**
   - `ChromaDB` 依赖和容器是预留的，主要服务于未来的 `GraphRAG`
   - 但当前实际的图谱构建、查询、可视化，走的是 **Apache `AGE`**

5. **当前仓库的 Docker 描述并不完全自洽**
   - 根目录 `docker-compose.yml` 里声明了 `frontend` 要构建 `./frontend`
   - 但仓库中**没有** `frontend/Dockerfile`
   - 所以“全项目 Docker 一键构建”在当前仓库状态下并不完整

6. **知识图谱功能依赖 Apache AGE 扩展**
   - 代码里明确执行 `LOAD 'age';`
   - 但仓库里没有提供安装 `AGE` 扩展的 Dockerfile 或初始化脚本
   - 这说明图谱功能的运行环境，当前更像是依赖现成 `AGE` 环境，而不是仓库内全自动搭建

7. **聊天展示代码用了 `marked / highlight.js / katex`**
   - 这些库在前端源码里被直接导入
   - 但 `frontend/package.json` 当前没有把它们显式声明出来
   - 这属于当前仓库里需要补齐的依赖描述问题

## 二、项目整体技术栈

这一节回答你最关心的问题：**项目整体用了什么技术，它们分别干什么，它们运行时怎么相互配合。**

### 2.2 各技术在项目中的具体应用场景

- **它是什么**：前端框架
- **你可以把它理解成**：专门写网页界面的主工具
- **项目里怎么用**：
  - `frontend/src/pages/` 里的所有页面都是 Vue 单文件组件
  - `frontend/src/components/` 里的聊天气泡、知识图谱组件也是 Vue 组件
- **具体场景**：
  - 写聊天页面
  - 写文档上传页面
  - 写知识图谱浏览页面

#### Vue Router

- **它是什么**：前端路由管理器
- **项目里怎么用**：
  - `frontend/src/router/index.ts`
- **具体场景**：
  - 访问 `/student/chat` 显示聊天页
  - 访问 `/student/ocr` 显示 OCR 页
  - 访问 `/student/kg-pipeline` 显示图谱构建页
  - 访问 `/student/knowledge` 显示图谱浏览页

#### Pinia

- **它是什么**：前端全局状态管理

  - `stores/auth.ts` 存 token 和用户
  - `stores/knowledge.ts` 管图谱列表、当前图谱、节点选择、搜索、删除
  - `stores/learning.ts` 目前只是预留
- **具体场景**：
  - 聊天页之外的组件也能读取登录态
  - 图谱页多个组件共享同一份图谱数据

#### Element Plus

- **它是什么**：前端 UI 组件库
- **项目里怎么用**：
  - 上传组件 `el-upload`
  - 表单组件 `el-form`
  - 提示组件 `el-alert` / `el-result`
  - 菜单布局 `el-menu`
- **具体场景**：
  - OCR 上传区
  - 图谱构建任务列表
  - 图谱页顶部工具栏、下拉选择器、删除确认框

#### ECharts

- **具体场景**：

## 三、项目运行时各技术的相互关系

这一节重点讲“技术之间怎么配合”。

### 3.1 聊天链路

```
Vue 页面 (Chat.vue)
  -> useChat.ts
  -> api/ai.ts 使用 fetch 建立 SSE（服务端流式推送）
  -> Backend ai_gateway.py
  -> AI chat.py
  -> LLMClient.stream()
  -> 外部/内网模型服务
  -> SSE 流回到前端
  -> ChatMessage.vue 渲染 Markdown / 数学公式 / 思考过程
```

对应关系：

- **Vue**：写聊天界面
- **Pinia**：这里没做主状态承载，聊天主要在 composable 内部维护
- **fetch + SSE**：让回答一段一段实时显示
- **FastAPI Backend**：做网关和服务隔离
- **FastAPI AI**：真正调用模型
- **LLMClient**：统一封装模型调用细节

### 3.2 OCR 链路

```
Vue 页面 (OcrParse.vue，OCR 解析页)
  -> useOcr.ts 负责提交和轮询
  -> frontend/api/ocr.ts
  -> backend/api/ocr_gateway.py
  -> ai/ocr/router.py
  -> AsyncTaskManager
  -> mineru OCR 服务
  -> 结果保存在 AI 服务输出目录
  -> 前端轮询状态并展示
```

对应关系：

- **Element Plus Upload**：负责前端文件上传交互
- **Axios**：提交表单和轮询任务状态
- **Backend 网关**：负责把多文件表单转发到 AI 服务
- **AI Task Manager**：负责异步任务编排
- **mineru**：负责真正文档解析

### 3.3 知识图谱构建链路

```
Vue 页面 (KgPipeline.vue，`Kg = Knowledge Graph`，知识图谱构建页)
  -> useKgPipeline.ts（知识图谱任务 composable）负责提交和轮询
  -> frontend/api/kgPipeline.ts
  -> backend/api/kg_gateway.py
  -> PostgreSQL 先创建 `kg_graphs`（知识图谱元数据表）记录
  -> Redis 保存 task_id 映射
  -> AI /kg/build 把任务压入 Redis 队列
  -> worker.py 取出任务
  -> tasks/kg_build.py
  -> KGPipeline.run_from_file()
  -> OCR 解析文档
  -> MarkdownChunker 切片
  -> KGExtractor（知识图谱抽取器）调 `LLM` 抽实体/关系
  -> GraphBuilder 用 NetworkX 合并
  -> AgeStorage 写入 Apache `AGE`
  -> 查询接口从 `AGE` 读图
  -> KnowledgeGraph.vue 用 ECharts 展示
```

这是当前仓库最有价值的一条主链路，因为它把很多技术串起来了：

- **Vue + Element Plus**：上传文件、看任务进度
- **FastAPI Backend**：建元数据、转发、同步任务状态
- **SQLAlchemy + PostgreSQL**：保存图谱记录
- **Redis**：做任务队列和状态缓存
- **FastAPI AI + Worker**：执行异步图谱构建
- **mineru**：先把文档转成可读文本
- **LLM**：从文本里抽实体和关系
- **NetworkX**：合并多块图结果
- **Apache AGE**：图数据最终存储和查询
- **ECharts**：前端可视化

### 3.4 当前“已接入”和“预留未接入”的技术关系

| 技术 | 当前状态 | 说明 |
|------|------|------|
| Vue / FastAPI / Redis / PostgreSQL / httpx / mineru / AGE / ECharts | **已接入主链路** | 这些是当前项目真正跑得起来的核心组合 |
| ChromaDB | 预留 | 未来给 GraphRAG 检索用 |
| PyTorch / PyG | 预留 | 未来给 GNN 推荐用 |
| OpenAI SDK | 依赖已装但当前代码未用 | 当前模型调用是自己用 `httpx` 封装的 |
| Monaco Editor | 包已装但当前页面没用 | 未来可能用于 SQL/代码编辑 |

## 六、项目实际运行架构

### 6.1 当前代码里能看出来的两套运行口径

#### 口径 A：远程服务器口径

`CLAUDE.md` 说明了远程环境大致是：

| 服务 | 端口 |
|------|------|
| 前端 Vite | 8000 |
| 后端 FastAPI | 8006 |
| AI FastAPI | 8001 |
| 聊天模型服务 | 8002 |
| OCR 模型服务 | 8004 |

这与 `frontend/vite.config.ts` 中的 `/api -> http://localhost:8006` 是一致的。

#### 口径 B：旧文档 / 本地口径

旧文档里曾经写过：

- 前端 5173
- 后端 8000
- AI 8001

这和当前 `vite.config.ts` 是不一致的。

### 6.2 因此当前最稳妥的理解

如果你按当前仓库代码本身来运行，**推荐把前端开发端口理解成 8000，后端理解成 8006，AI 理解成 8001**，因为这是和现有代理配置一致的。

### 6.3 Docker 现状

| 文件 | 现状 |
|------|------|
| `docker-compose.middleware.yml` | 相对可信，可用于启动 postgres / redis / chromadb |
| `docker-compose.yml` | 不完全可直接用，原因是 `frontend/Dockerfile` 缺失 |

### 6.4 图谱功能的额外环境要求

知识图谱功能不仅需要：

- PostgreSQL
- Redis
- AI 服务
- OCR 模型服务
- LLM 抽取模型

还需要：

- **Apache AGE 扩展环境**

因为 `AgeStorage` 中会直接执行：

```sql
LOAD 'age';
SET search_path TO ag_catalog, public;
```

所以如果运行环境里没有 AGE，这条知识图谱主链路就跑不通。

## 九、最终判断

### 9.1 这个项目现在最准确的定位

它不是一个“已经全部完成的教学平台”，而是一个：

> **以聊天、OCR（文档文字识别/文档解析）、知识图谱构建为核心原型的 AI 教学平台雏形**

### 9.2 它已经有价值的地方

真正有价值、而且不是空壳的部分有三点：

1. **三服务分层已经清楚**
2. **AI 异步任务体系已经成型**
3. **知识图谱构建流水线已经从前端贯通到图数据库**

### 9.3 它还缺什么

它还缺的是“平台层业务”的完成：

1. 用户系统
2. 学习系统
3. 题目系统
4. 教师系统
5. `GraphRAG`
6. `GNN`

也就是说：

- **智能底座有雏形**
- **平台业务还在建设中**
